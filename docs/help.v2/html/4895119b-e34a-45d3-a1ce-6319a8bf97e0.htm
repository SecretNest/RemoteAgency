Ping and Cluster

Ping test is designed for testing the delay and the connectivity between proxy and remote service wrapper.

Ping

Ping test can be initiated through proxy by calling Ping or TryPing methods, making a message generated and sent to the target specified when proxy is created. Methods end after a response received with test result calculated.

Ping and TryPing methods include: Ping(Guid, Guid, Guid), Ping(Guid, TimeSpan, Guid, Guid), TryPing(Guid, TimeSpan, TimeSpan, Guid, Guid, Exception), TryPing(Guid, TimeSpan, Guid, Guid, Exception) and TryPing(Guid, TimeSpan, TimeSpan, Guid, Guid, Exception).

Real Target in Cluster

In particular, ping methods contain the target site id and instance id as output parameters. They can be used to get the real target from a virtual target.

While creating instance of proxy, a default target site id and instance id need to be specified. After initialized, most operations started by proxy will use this as the target of the message, aka will ask the service wrapper with the site id and instance id to access the asset required. The only one kind of message not use this as target, is the event removing message. It will always send the message to the sender of the response message of related event adding message.

When the message routing system has load balancing design, like Remote Hub, by creating a cluster based on many real sites, it allows to distribute the proxy-initiated operations among several servers which contains service wrapper for servicing. If the target site of one proxy is set to a virtual one, all accessing may not be sent to the same real service wrapper. It’s good for load balancing and fail over but may not be friendly with stateful service designing. By creating another proxy using the target grabbed from ping test result, the only one service wrapper from the virtual group is selected and specified.
