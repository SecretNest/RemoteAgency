1: Method Overloading and Customized Asset Name

Remote Agency supports method overloading like coding elsewhere.

Prerequisites
Before using this example code, an app need to be created first. Follow this doc to prepare that first.
Brief

Remote Agency supports method overloading in the same as the dotnet language specification.

Each asset need a unique name which is generated by default. User can specify the asset name using CustomizedAssetNameAttribute but an exception will be thrown when name conflicts.

Steps

  1. Create an interface that contains all the assets used for communication.

    C#
    VB
    C++
    F#
    Copy
    public interface ITest1
{
    int Add(int a, int b);

    //When unmark this line below, an exception about asset naming conflicts will be thrown.
    //[CustomizedAssetName("AddDouble")]
    float Add(float a, float b);

    [CustomizedAssetName("AddDouble")]
    double Add(double a, double b);
}
    Public Interface ITest1
  Function Add(a As Integer, b As Integer) As Integer

  'When unmark this line below, an exception about asset naming conflicts will be thrown.
  '<CustomizedAssetName("AddDouble")>
  Function Add(a As Single, b As Single) As Single

  <CustomizedAssetName("AddDouble")>
  Function Add(a As Double, b As Double) As Double
End Interface

  Implements ITest1

  Public Function Add(a As Integer, b As Integer) As Integer Implements ITest1.Add
    Return a + b
  End Function

  Public Function Add(a As Single, b As Single) As Single Implements ITest1.Add
    Return a + b
  End Function

  Public Function Add(a As Double, b As Double) As Double Implements ITest1.Add
    Return a + b
  End Function
End Class

  3. Place this code to a method, main method for example.

    C#
    VB
    C++
    F#
    Copy
    //test router
var router = new RemoteAgencyRouter<byte[], object>();

//Server
var originalService = new Server1();
using var serverRemoteAgencyInstance = RemoteAgencyBase.CreateWithBinarySerializer(true);
router.AddRemoteAgencyInstance(serverRemoteAgencyInstance);
var serverSiteId = serverRemoteAgencyInstance.SiteId;
var serviceWrapperInstanceId = serverRemoteAgencyInstance.CreateServiceWrapper(originalService);

//Client
using var clientRemoteAgencyInstance = RemoteAgencyBase.CreateWithBinarySerializer(true);
router.AddRemoteAgencyInstance(clientRemoteAgencyInstance);
var clientProxy = clientRemoteAgencyInstance.CreateProxy<ITest1>(serverSiteId, serviceWrapperInstanceId).ProxyGeneric;

//Run test
Console.WriteLine("Add(int): 1 + 2");
Console.WriteLine(clientProxy.Add(1, 2));

Console.WriteLine("Add(float): 1.1 + 2.2");
Console.WriteLine(clientProxy.Add(1.1f, 2.2f));

Console.WriteLine("Add(double): 1.1 + 2.2");
Console.WriteLine(clientProxy.Add(1.1, 2.2));

Console.Write("Press any key to continue...");
Console.ReadKey(true);
Console.WriteLine();

    When creating proxy, target site id and service wrapper instance id need to be specified.

    Site id is used to locate the server. In a real scenario, servers are usually deployed into different applications even computers. Message routing component, like Remote Hub, need to use the site id to deliver messages generated from proxy to the right server instance.

    Service wrapper instance id is used to locate the service object which contains user code. This is used by Remote Agency to deliver the message to the right service object registered by CreateServiceWrapper (or CreateServiceWrapper).

    No such id is required when creating service wrapper because the response messages are sent to the source of the request.
