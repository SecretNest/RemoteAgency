Exception Redirect

The way to deal with the exception thrown when accessing by remote site.

Exception redirect

Normally, when an exception is thrown in the code accessed by remote site, only the caller will receive the exception, but the site which running this code won't be affected.

You can change this behavior by each asset if you need.

Using LocalExceptionHandlingAttribute on the asset (method, event and property) of interface, you can choose how to deal the exception:

  • Suppress: Like not using this attribute.

  • Throw: Throw the exception in place after sending it back to caller. Caution: You may not easy to catch this exception.

  • Redirect: Redirect the exception after sending it back to caller.

All exception redirected, will be transferred by raising an event RedirectedExceptionRaised. If this event is not handled, it will be ignored.

Specially, if there is any error occurred while serializing processing of preparing the error message feedback in service wrapper when requested asset cannot be found, LocalExceptionHandling setting from the only or the first of all related interfaces will be applied.

Service wrapper asset not found

When all conditions below are all matched:

  • Service wrapper is created based on multiple interfaces,

  • The asset requested by the remote object (proxy) cannot be found, and

  • While preparing the feedback message for asset cannot be found, any exception occurred in serializing processing.

The setting of the first interface specified with service wrapper creating process will be used in this (serializing exception of preparing message for asset requested cannot be found) error handling.
